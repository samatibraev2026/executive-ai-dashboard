**Управленческий AI-дашборд для руководителей компании**

## 1. Обзор продукта

**Executive AI Dashboard** — единый экран контроля для топ-менеджмента компании, объединяющий в одном интерфейсе состояние ИТ-систем, инциденты, риски, проекты, качество данных и AI-инициативы. Продукт даёт руководителям возможность за несколько секунд понять, что требует их внимания, без необходимости запрашивать отчёты у разных команд.

Дашборд ориентирован на пять управленческих ролей — **CEO, CIO, CDO, CTO, CAIO** — и предоставляет каждой роли отдельную, релевантную её зоне ответственности страницу, а также общую сводную главную страницу (Executive Summary).

### 1.1 Проблема, которую решает продукт

Сегодня информация о состоянии ИТ, рисках, проектах и AI-инициативах разбросана по разным системам и командам (мониторинг, трекеры инцидентов, реестры рисков, проектные офисы, BI-отчёты). Руководители тратят время на сбор данных из разных источников вместо принятия решений. Нет единой точки, где видно:

- какие системы работают нормально, а какие деградируют;
- сколько открыто критических инцидентов и кто за них отвечает;
- какие риски (технологические, data, кибербезопасность, AI) требуют решения;
- какие проекты отстают по срокам или бюджету;
- какой реальный эффект приносят AI-инициативы;
- какие вопросы нужно решить именно руководителю, а не команде.

### 1.2 Решение

Веб-приложение с единым набором данных, ролевыми срезами и сквозной системой статусов (🟢 норма / 🟡 внимание / 🔴 критично), которое:

- агрегирует данные по 7 ключевым блокам (системы, инциденты, риски, проекты, данные, AI, рекомендации);
- предоставляет фильтрацию по периоду, системе, статусу и критичности;
- даёт каждой роли свой набор KPI и таблиц, релевантный её мандату;
- работает локально или как статическая веб-страница, без сложной инфраструктуры.

## 2. Цели и задачи

### 2.1 Бизнес-цели

1. Сократить время, необходимое руководителю для оценки состояния цифрового контура компании — с часов (сбор отчётов) до минут (один экран).
2. Повысить прозрачность статуса ИТ-систем, рисков и проектов для совета директоров и топ-менеджмента.
3. Создать единый язык статусов (🟢/🟡/🔴) для всех цифровых и AI-инициатив компании.
4. Дать CAIO инструмент для демонстрации измеримого эффекта от AI-портфеля.

### 2.2 Продуктовые цели (для прототипа)

1. Покрыть все 9 содержательных блоков из исходного технического задания (системы, инциденты, риски, проекты, данные, AI, рекомендации, фильтры, навигация).
2. Реализовать 5 ролевых дашбордов + 1 общую главную страницу.
3. Обеспечить рабочий прототип, который можно запустить **за одну команду** (Streamlit) или **открыть напрямую в браузере** (HTML/JS-версия), без сложных зависимостей.
4. Весь интерфейс — на русском языке.

### 2.3 Метрики успеха (для будущей продуктовой версии)

| Метрика | Целевое значение |
|---|---|
| Время до выявления критической проблемы руководителем | < 1 минуты с момента открытия дашборда |
| Доля руководителей, использующих дашборд еженедельно | > 80% целевой аудитории |
| Среднее число кликов до нужного раздела | ≤ 2 |
| Доля инцидентов/рисков, отслеженных через дашборд, а не через ручные отчёты | рост во времени |

## 3. Целевая аудитория

| Роль | Зона ответственности в продукте | Ключевые вопросы, на которые роль ищет ответ |
|---|---|---|
| **CEO** | Стратегический обзор всей компании | Стабилен ли цифровой контур? Какие решения нужны от меня? Какой эффект от AI и куда уходит бюджет? |
| **CIO** | ИТ-системы, доступность, SLA, ИТ-инциденты, ИТ-проекты | Какие системы под угрозой? Где нарушается SLA? Что в портфеле ИТ-проектов отстаёт? |
| **CDO** | Качество данных, DWH, интеграции, data-риски | Каким данным можно доверять? Где разрывы в интеграциях? Какие data-риски критичны? |
| **CTO** | Инфраструктура, кибербезопасность, технологические риски | Устойчива ли инфраструктура? Какие кибер-риски требуют действий? Что с критическими инцидентами безопасности? |
| **CAIO** | AI use-cases, зрелость внедрения, эффект, AI-риски | Какие AI-инициативы работают и масштабируются? Где AI создаёт риск (bias, дрифт, неконтролируемое использование)? |

Все пользователи — топ-менеджеры, не технические специалисты по умолчанию: интерфейс должен быть понятным без обучения, с минимумом терминологии и понятной цветовой индикацией.

## 4. Функциональные требования

### 4.1 Главная страница (Executive Summary)

- Сводные KPI верхнего уровня по всей компании: доля систем в норме, открытые инциденты (в т.ч. критические), риски, требующие внимания, проекты с отклонением, суммарный эффект от AI.
- Таблица состояния ИТ-систем с цветовой индикацией.
- Топ-5 рисков, инциденты, проекты — в сокращённом виде.
- Сводка по качеству данных и AI-инициативам.
- Список управленческих рекомендаций, отсортированный по приоритету.

### 4.2 Ролевые дашборды

Каждая ролевая страница (CEO / CIO / CDO / CTO / CAIO) содержит:

1. **Блок KPI** (4 показателя), специфичных для роли.
2. **Таблицы/карточки**, отфильтрованные по зоне ответственности роли (например, у CIO — только ИТ-проекты, у CDO — только риски категории «Данные»).
3. Применение глобальных фильтров (период, система, статус, критичность) там, где это релевантно роли.

Детализация по ролям — см. раздел 3 (таблица) и блок 4.3 ниже.

### 4.3 Содержательные блоки данных

| Блок | Поля / атрибуты | Источник статуса |
|---|---|---|
| **ИТ-системы** | название, домен, критичность, доступность %, SLA-цель, владелец, дата обновления | статус по доступности относительно пороговых значений (99.5% / 98.0%) |
| **Инциденты** | ID, заголовок, система, критичность (критический/высокий/средний/низкий), статус (открыт/в работе/закрыт), даты создания/закрытия, владелец, время реакции | статус = критичность + открытость |
| **Риски** | название, категория (технологический/данные/кибербезопасность/AI), вероятность, влияние, владелец, план митигации | уровень = вероятность × влияние |
| **Проекты** | название, ответственная роль, прогресс %, бюджет план/факт, отклонение %, срок | статус по прогрессу и отклонению бюджета |
| **Качество данных** | источник, полнота %, точность %, своевременность %, индекс качества, статус интеграции | статус по среднему индексу качества |
| **AI-инициативы** | use-case, домен, зрелость (пилот → масштабируется), % внедрения, эффект (млн руб/мес), владелец | статус по эффекту и % внедрения |
| **Рекомендации** | текст, ответственная роль, приоритет (высокий/средний/низкий), срок | приоритет напрямую транслируется в статус |

### 4.4 Фильтры

- **Период**: последние 7 / 30 / 90 дней, весь период.
- **Система**: выбор конкретной ИТ-системы из общего списка.
- **Статус**: норма / внимание / критично.
- **Критичность**: низкая / средняя / высокая.

Фильтры применяются к таблицам систем, инцидентов и (где применимо) рисков на текущей открытой странице.

### 4.5 Навигация

- Переключение между главной страницей и пятью ролевыми дашбордами одним кликом/тапом.
- Текущий активный раздел визуально выделен.

### 4.6 Цветовая индикация статусов (сквозной принцип)

| Статус | Цвет | Значение |
|---|---|---|
| 🟢 Норма | зелёный | показатель в пределах нормы, действий не требуется |
| 🟡 Внимание | жёлтый/янтарный | показатель приближается к пороговому значению, нужен мониторинг |
| 🔴 Критично | красный | требуется управленческое решение или немедленное действие |

## 6. Дизайн и UX

Сделаны две стилистические итерации:

### 6.1 Итерация 1 — деловой корпоративный стиль
Светлый фон, нейтральная цветовая палитра, табличные карточки KPI с цветной полосой статуса слева, шрифтовая пара Fraunces (заголовки) + Inter (текст) + IBM Plex Mono (числа/KPI).

### 6.2 Итерация 2 — стиль apple.com (текущая версия)
По прямому запросу заказчика интерфейс переоформлен в эстетике apple.com:

- **Типографика**: системный шрифт `-apple-system` / `SF Pro` с фолбэком на Inter; крупные жирные заголовки с плотным трекингом (letter-spacing от −0.02 до −0.03em).
- **Навигация**: «стеклянная» закреплённая панель сверху (`backdrop-filter: blur`), полупрозрачный фон.
- **Hero-блок**: крупный заголовок и подзаголовок на светло-серой подложке (#F5F5F7) — приём, характерный для продуктовых страниц Apple.
- **Переключатель ролей**: сегментированный контрол (`segmented control`) — закруглённая «таблетка» с белой активной вкладкой, как в интерфейсах Apple.
- **KPI-карточки**: крупное жирное число, статус обозначен небольшой цветной точкой (без цветных полос-«рельсов»), как в карточках характеристик на apple.com.
- **Цветовая палитра статусов**: системные цвета в духе iOS (зелёный `#1F8A44`, оранжевый `#C8780A`, красный `#D70015`), акцентный синий `#0071E3`.
- **Карточки/таблицы**: увеличенные радиусы скругления (18px), мягкие тени, минимум визуального шума.
- **Анимация**: плавное появление контента (fade + lift) при переключении вкладок, с уважением `prefers-reduced-motion`.

## 7. Техническая архитектура

Подготовлены **два независимых прототипа**, решающих одну и ту же задачу разными средствами:

### 7.1 Прототип А — Python + Streamlit

Для случаев, когда дашборд будет подключаться к реальным backend-источникам данных (БД, DWH, мониторинг, Jira/SOC).

```
ai_dashboard/
├── app.py                  # Главная страница (Executive Summary)
├── requirements.txt        # streamlit, pandas, numpy
├── README.md
├── data/
│   └── demo_data.py        # Генерация демо-данных по всем блокам
├── utils/
│   └── helpers.py          # KPI-карточки, фильтры, цветовая индикация
└── pages/                  # Ролевые дашборды (Streamlit Multipage)
    ├── 1_👑_CEO.py
    ├── 2_🖥️_CIO.py
    ├── 3_🗄️_CDO.py
    ├── 4_🛠️_CTO.py
    └── 5_🤖_CAIO.py
```

Запуск: `pip install -r requirements.txt && streamlit run app.py`.

### 7.2 Прототип Б — автономный HTML/CSS/JavaScript

Для мгновенного доступа без установки чего-либо: один HTML-файл, открываемый прямо в браузере или размещаемый на статическом хостинге / внутреннем портале.

- Чистый JavaScript (без фреймворков), детерминированная генерация демо-данных через сидированный псевдослучайный генератор (воспроизводимые данные между перезагрузками).
- Рендеринг через шаблонные строки и `innerHTML`, без сборщиков и зависимостей.
- Стилизация на CSS-переменных (токены темы) для единообразного оформления.
- Шесть вкладок (Главная + 5 ролей), фильтры, KPI, таблицы с цветовой индикацией — полный паритет функциональности с прототипом А.

### 7.3 Принцип расширения на реальные данные

Для перехода от демо-данных к реальным источникам:

1. Заменить функции-генераторы (`genSystems`, `genIncidents`, … в JS или `get_systems()`, … в Python) на запросы к реальным источникам (БД, API мониторинга, трекер инцидентов, реестр рисков, проектный офис, AI-платформа), сохранив названия и типы полей.
2. Остальной код интерфейса (KPI, таблицы, фильтры, ролевые страницы) не требует изменений, так как работает с унифицированной структурой данных.

## 9. Ограничения и допущения текущей версии (прототип)

- Все данные — синтетические демонстрационные, не подключены к реальным источникам.
- Фильтр «Период» применяется к датам инцидентов; не все блоки имеют временное измерение в демо-данных.
- Авторизация и разграничение доступа по ролям не реализованы — все страницы доступны всем пользователям.
- Графики/тренды (динамика во времени) не реализованы — только табличные представления и точечные KPI.
- Экспорт в PDF/Excel не реализован.
